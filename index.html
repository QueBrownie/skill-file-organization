Organizing Skill Files for

Follow the flow: structure → loading → orchestration → principles → pitfalls. Each step builds on the last.

Step 1 — Know the Structure

Canonical Skill Directory

Every skill lives in its own folder. Only SKILL.md is required. Three official bundled resource types live alongside it.

my-skill/ ├── SKILL.md (required) │ ├── YAML frontmatter ← name + description │ └── Markdown body ← instructions │ ├── scripts/ (optional) │ └── build_report.py ← deterministic / repetitive tasks │ ├── references/ (optional) │ └── aws.md ← docs loaded into context as needed │ └── assets/ (optional) └── template.html ← templates, icons, fonts

Large orchestration skills often add more directories. Common extensions include agents/ for subagent role definitions and evals/ for test cases.

SKILL.md — The Brain

The only file Claude reads when a skill triggers. Everything else is pulled in on demand via explicit pointers.

--- name: daily-prep description: > Run the full morning prep sequence. Use whenever the user says "daily prep", "start my day", etc. --- # Daily Prep — Orchestrator ## Step 1: Get Context Fetch calendar, find calls... ## Step 2: Run Inbox Triage Follow `inbox-triage` skill...
Key rule: Keep SKILL.md under ~500 lines. If you're approaching the limit, offload detail into references/ with clear pointers about when to read them.
The structure is set. Now, how does Claude actually load all of this?
Step 2 — Understand the Loading Model
1
Always in Context

Metadata

name + description from YAML frontmatter. Always visible in Claude's skill list (~100 words). This is the trigger mechanism — it determines whether the skill fires at all.

Tip: Make descriptions "pushy" — list trigger phrases and edge cases. Under-triggering is the #1 problem.

2
On Trigger

SKILL.md Body

Full workflow instructions, output format specs, and pointers to bundled resources. Loaded into context only when the skill triggers. Target <500 lines.

Tip: Explain the "why" behind instructions — Claude generalizes better from reasoning than rigid rules.

3
As Needed

Bundled Resources

Scripts, references, and assets. Loaded only when SKILL.md explicitly tells Claude to read them. Scripts can execute without loading into context — zero cost.

Tip: Unlimited size. Great for domain variants and large templates.

CHEAP
EXPENSIVE
Loading is progressive: metadata is always there, the body loads on trigger, resources load on demand.
Step 3 — Choose an Orchestration Pattern

Pattern A: Skill Tree

One orchestrator calls sub-skills in sequence. Each sub-skill also works independently.

daily-prep
orchestrator
inbox-triage
slack-review
call-prep
## Step 1: Get Context Fetch calendar, identify external calls. ## Step 2: Run Inbox Triage Follow the `inbox-triage` skill fully. Capture the full output. ## Step 3: Run Slack Review Follow the `slack-review` skill fully. Capture the full output. ## Step 4: Run Call Prep For each meeting from Step 1, follow `call-prep` skill fully. ## Step 5: Compose & Post Combine all outputs into one message.

Pattern B: Domain Variants

One skill, multiple domains. SKILL.md routes to the right reference file based on context.

cloud-deploy/ ├── SKILL.md ← shared workflow + routing └── references/ ├── aws.md ← AWS-specific steps ├── gcp.md ← GCP-specific steps └── azure.md ← Azure-specific steps

Claude reads only the relevant reference — keeping context lean.

Pattern C: Subagent Delegation

For parallel or isolated work, define agent roles in an agents/ directory.

skill-creator/ ├── SKILL.md └── agents/ ├── grader.md ← evaluates assertions ├── comparator.md ← blind A/B comparison └── analyzer.md ← analyzes results
You've picked a pattern. Now make sure it holds up with these principles.
Step 4 — Follow Core Principles

Explain the Why

Instead of rigid ALWAYS/NEVER rules, explain the reasoning. Claude generalizes better from rationale than from commands. If you're writing in all-caps, reframe as a justification.

Keep It Lean

SKILL.md is context-window real estate. Move domain detail → references/, repetitive logic → scripts/, agent personas → agents/. Reference them clearly with "when to read" guidance.

Sub-Skills Stand Alone

Every sub-skill in an orchestration tree must be independently triggerable. A user should be able to run inbox-triage on its own or as part of daily-prep. Never make a skill dependent on being called by a parent.

Generalize, Don't Overfit

Skills are used thousands of times across many different prompts. When iterating, encode the underlying principle — not a narrow fix for one specific test case. If a change only works for one example, find the deeper pattern.

And finally — some quick heuristics to keep in your pocket.
Step 5 — Avoid Common Pitfalls
Do This
  • Use progressive disclosure — metadata → body → resources
  • Bundle repeated scripts — if subagents keep writing the same helper code, make it shared
  • Write pushy descriptions — list every trigger phrase a user might say
  • Handle partial failure — orchestrators degrade gracefully, not catastrophically
  • Include input/output examples for non-obvious formats
Don't Do This
  • Don't exceed ~500 lines in SKILL.md — refactor into references
  • Don't use vague descriptions — "a helpful skill for data" won't trigger
  • Don't hardcode context — avoid baking in names, dates, or values that change
  • Don't duplicate logic — extract shared steps into a sub-skill or reference
  • Don't litter with MUST/NEVER — explain the reasoning instead
Scripts If all test runs independently write the same helper script, bundle it in scripts/. Save every future run from reinventing the wheel.
Large References For reference files over 300 lines, include a table of contents at the top so Claude can navigate efficiently.
Error Handling If a sub-skill fails, still run the others. Note the failure in output. A partial result beats no result.
Triggering Under-triggering is the #1 skill problem. Be aggressive with description phrases — cast a wide net.

Quick-Reference Cheat Sheet

Sizing

Description: ~100 words
SKILL.md body: <500 lines
Reference files: unlimited
Ref >300 lines: add a TOC
Scripts: no context cost

When to Create Sub-Files

scripts/ → deterministic logic
references/ → domain-specific docs
assets/ → templates, images, fonts
agents/ → subagent roles (extended)
evals/ → test cases (extended)

Orchestrator Checklist

✓ Gather shared context first
✓ Call sub-skills by name
✓ Capture each sub-skill's output
✓ Handle partial failures gracefully
✓ Assemble and deliver final result
