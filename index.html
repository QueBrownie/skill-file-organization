What is a Skill?

Think of it like a recipe

A skill is a set of instructions you write for Claude, saved in a file. Whenever you say something that matches the skill, Claude reads those instructions and follows them — every time, consistently.

📋

Without a skill: you explain what you want from scratch every conversation.
With a skill: you say "triage my inbox" and Claude already knows exactly what to do.

What can you make a skill do?

Anything you find yourself asking Claude to do repeatedly. Some examples:

  • Check your email and surface what needs replies
  • Summarize your Slack channels each morning
  • Prep you before a sales or client call
  • Write a weekly status report from your notes
  • Review code before it goes out
💡

If you've copy-pasted the same instructions to Claude more than twice, that's a skill waiting to be written.

Ready to make one? Here's all you need.
How to Create Your First Skill

You only need one file

A skill lives in a folder with a single required file called SKILL.md. That's it. The file has two parts — a short header at the top, and your instructions below.

--- (this marks the start of the header) name: inbox-triage description: Check my Gmail and surface what needs replies. Use when I say "triage inbox", "check email", or "what needs replies". --- (this marks the end of the header) Your instructions go here. Write them like you're explaining the job to a smart helper. Step 1: Open Gmail and find unread emails. Step 2: For each one, decide if it needs a reply, follow-up, or can be skipped. Step 3: Show me a summary with the most urgent items first.

Writing good instructions

Claude is smart — you don't need to be super precise. Write your instructions the way you'd explain the job to a new but capable colleague.

1
Say what the goal is

What should Claude accomplish by the end? Start there.

2
Break it into steps

Number them. Claude follows numbered steps reliably.

3
Describe what "done" looks like

Tell Claude what the output should look like — a list, a summary, a message draft, etc.

4
Explain the why, not just the what

Instead of "always flag emails from VPs", say "flag emails from VPs because they usually need same-day replies."

The most important part: making it trigger.
Making Your Skill Trigger

The description is the trigger

The description in your skill's header is how Claude decides when to use it. Claude reads your description and asks: "does this match what the user is asking for?"

The more trigger phrases you include, the more reliably it fires. List the different ways you might ask for the same thing.

🔍

The #1 reason skills don't work is a vague description. Claude can't read your mind — it can only match what you wrote.

What good trigger phrases look like

These are too vague:

"helps with email" "morning tasks" "useful for calls"

These will actually trigger:

"triage my inbox" "check my email" "what needs replies" "clear my inbox" "what's waiting on me"
✍

Think about all the different ways you might ask for this on a busy morning. List all of them in your description.

That's the core of it. A few quick tips to make yours great.
Quick Tips for Better Skills
Do This
  • One skill, one job. Don't try to cram everything into one skill. A focused skill works better than a sprawling one.
  • Test it right after writing. Ask Claude in a few different ways. If it doesn't fire, add more phrases to the description.
  • Say "done" clearly. Tell Claude what the final output should look like — a bullet list, a Slack message, a summary table, etc.
  • Update it when your process changes. Skills aren't set-and-forget. If you start doing things differently, update the steps.
Avoid This
  • Vague descriptions. "A skill for morning tasks" won't fire. Be specific about what triggers it.
  • Hardcoding things that change. Don't write "check emails from John" — John might leave. Write "check emails from my manager."
  • Instructions that contradict each other. If step 3 says one thing and step 7 says the opposite, Claude will guess. Pick one.
  • Too many rules without reasons. "Always do X" is weaker than "Always do X because Y." The reason helps Claude apply it correctly.
First skill idea What's something you've explained to Claude more than twice? That's your first skill. Start there, not with something complex.
Not triggering? Open your SKILL.md and add more phrases to the description. That's almost always the fix.

Organizing Skill Files for
Large Orchestration

Follow the flow: structure → loading → orchestration → principles → pitfalls → descriptions → testing → power patterns → right tool. Each step builds on the last.

Step 1 — Know the Structure

Canonical Skill Directory

Every skill lives in its own folder. Only SKILL.md is required. Three official bundled resource types live alongside it.

my-skill/ ├── SKILL.md (required) │ ├── YAML frontmatter ← name + description (the trigger) │ └── Markdown body ← full workflow instructions │ ├── scripts/ (optional) │ └── build_report.py ← deterministic / repetitive logic │ ├── references/ (optional) │ └── aws.md ← domain docs, loaded on demand │ ├── assets/ (optional) │ └── template.html ← templates, icons, fonts │ ├── agents/ (optional, extended) │ └── grader.md ← subagent role definitions │ └── evals/ (optional, extended) └── test_cases.md ← test prompts + expected behaviors

Large orchestration skills often add agents/ for subagent role definitions and evals/ for test cases — both are extensions of the core structure, not official types.

SKILL.md — A Complete Example

The only file Claude reads when a skill triggers. Everything else is pulled in on demand via explicit pointers in the body.

--- name: daily-prep description: > Run the full morning prep sequence. Use whenever the user says "daily prep", "start my day", "morning briefing", "what's on my plate", or asks for a summary of their day ahead. --- # Daily Prep — Orchestrator ## Step 1: Gather Shared Context Fetch today's calendar. Identify external calls. Note any blockers visible from prior context. ## Step 2: Run Inbox Triage Follow the `inbox-triage` skill fully. Capture the complete output before moving on. ## Step 3: Run Slack Review Follow the `slack-review` skill fully. Capture the complete output before moving on. ## Step 4: Prep Each Call For each external meeting from Step 1, follow the `call-prep` skill fully. ## Step 5: Compose & Deliver Combine all outputs into one structured briefing. Post to the user's daily thread. ## Output Format Use the template in `assets/daily-brief.md`.
Key rule: Keep SKILL.md under ~500 lines. If you're approaching the limit, offload detail into references/ with clear pointers about when to read them.
The structure is set. Now, how does Claude actually load all of this?
Step 2 — Understand the Loading Model
1
Always in Context

Metadata

name + description from YAML frontmatter. Always visible in Claude's skill list (~100 words). This is the trigger mechanism — it determines whether the skill fires at all.

Tip: Make descriptions "pushy" — list trigger phrases and edge cases. Under-triggering is the #1 problem.

2
On Trigger

SKILL.md Body

Full workflow instructions, output format specs, and pointers to bundled resources. Loaded into context only when the skill triggers. Target <500 lines — this is real context-window space, so every line has a cost.

Tip: Explain the "why" behind instructions — Claude generalizes better from reasoning than rigid rules.

3
As Needed

Bundled Resources

Scripts, references, and assets. Loaded only when SKILL.md explicitly tells Claude to read them. Scripts are executed via the Bash tool — Claude runs them and gets the output, but the script source never enters context. A 500-line script costs zero context tokens.

Tip: Unlimited size. Great for domain variants and large templates.

CHEAP
EXPENSIVE
Loading is progressive: metadata is always there, the body loads on trigger, resources load on demand.
Step 3 — Choose an Orchestration Pattern

Pattern A: Skill Tree

One orchestrator calls sub-skills in sequence. Each sub-skill also works independently.

daily-prep
orchestrator
inbox-triage
slack-review
call-prep
## Step 2: Run Inbox Triage Follow the `inbox-triage` skill fully. Capture the full output before continuing. ## Step 3: Run Slack Review Follow the `slack-review` skill fully. Capture the full output before continuing. ## Step 4: Run Call Prep For each meeting from Step 1, follow `call-prep` skill fully. ## Step 5: Compose & Post Combine all outputs into one message.

Pattern B: Domain Variants

One skill, multiple domains. SKILL.md routes to the right reference file based on context — keeping context lean by loading only what's needed.

cloud-deploy/ ├── SKILL.md ← shared workflow + routing logic └── references/ ├── aws.md ← AWS-specific steps ├── gcp.md ← GCP-specific steps └── azure.md ← Azure-specific steps
## Step 1: Detect Platform Check for aws/gcp/azure config in context. ## Step 2: Load Domain Docs If AWS: read `references/aws.md` If GCP: read `references/gcp.md` If Azure: read `references/azure.md`

Pattern C: Subagent Delegation

For parallel or isolated work, define agent roles in an agents/ directory. The orchestrator reads each agent file and spawns a subagent with that persona — each starts fresh with no shared context bleed.

skill-creator/ ├── SKILL.md └── agents/ ├── grader.md ← evaluates assertions independently ├── comparator.md ← blind A/B comparison, no prior bias └── analyzer.md ← synthesizes all results
## Step 1: Grade Assertions Read `agents/grader.md` for grader persona. Spawn a subagent using that persona. Pass each assertion independently. Collect results before continuing. ## Step 2: Compare Outputs (blind) Read `agents/comparator.md`. Spawn subagent — pass both outputs without labels. Collect ranking + rationale. ## Step 3: Synthesize Read `agents/analyzer.md`. Spawn final subagent with all prior results.
Why isolation matters: Each subagent starts fresh — no context bleed between grading, comparing, and analyzing. This prevents earlier results from biasing later ones.
You've picked a pattern. Now make sure it holds up with these principles.
Step 4 — Follow Core Principles

Explain the Why

Instead of rigid ALWAYS/NEVER rules, explain the reasoning. Claude generalizes better from rationale than from commands. If you're writing in all-caps, reframe as a justification.

Keep It Lean

SKILL.md is context-window real estate. Move domain detail → references/, repetitive logic → scripts/, agent personas → agents/. Reference them clearly with "when to read" guidance.

Sub-Skills Stand Alone

Every sub-skill in an orchestration tree must be independently triggerable. A user should be able to run inbox-triage on its own or as part of daily-prep. Never make a skill dependent on being called by a parent.

Generalize, Don't Overfit

Skills are used thousands of times across many different prompts. When iterating, encode the underlying principle — not a narrow fix for one specific test case. If a change only works for one example, find the deeper pattern.

Principles in place — now avoid the common traps.
Step 5 — Avoid Common Pitfalls
Do This
  • Use progressive disclosure — metadata → body → resources
  • Bundle repeated scripts — if subagents keep writing the same helper code, make it shared
  • Write pushy descriptions — list every trigger phrase a user might say
  • Handle partial failure — orchestrators degrade gracefully, not catastrophically
  • Include input/output examples for non-obvious formats
  • Add a TOC to reference files over 300 lines
Don't Do This
  • Don't exceed ~500 lines in SKILL.md — refactor into references
  • Don't use vague descriptions — "a helpful skill for data" won't trigger
  • Don't hardcode context — avoid baking in names, dates, or values that change
  • Don't duplicate logic — extract shared steps into a sub-skill or reference
  • Don't litter with MUST/NEVER — explain the reasoning instead
  • Don't make sub-skills parent-dependent — they must run standalone
Now let's tackle the #1 failure point: descriptions that don't trigger.
Step 6 — Write Descriptions That Actually Trigger

The Trigger Mechanism

Claude scans all skill descriptions on every turn to decide which skill (if any) to fire. The description is your only lever for controlling when your skill runs. Claude matches on meaning, not keywords — but coverage matters.

Mental model: Write your description as if briefing a smart colleague who has never seen your skill. What would they need to know to recognize when to use it?

A strong description has three parts:

  • What it does — one crisp sentence
  • Trigger phrases — exact things the user might say
  • Edge cases — non-obvious situations where it should still fire
Over-triggering fix: If your skill fires too broadly, add exclusion notes — "use for X, not for Y" helps Claude discriminate between similar skills.

Before & After

Weak Description Strong Description Why It Matters
"A helpful skill for doing tasks with data" "Run full data pipeline analysis. Use when user says 'analyze data', 'run the pipeline', 'check the data', 'data report', or mentions wanting a data summary." Vague descriptions almost never trigger. Claude needs enough signal to match intent.
"Helps with email" "Triage Gmail inbox for action items. Use when user says 'triage inbox', 'check my email', 'what emails need replies', 'clear my inbox', or asks what's waiting on them." Concrete trigger phrases dramatically increase coverage — users rarely say exactly what you expect.
"Morning routine skill" "Run full morning prep: calendar, inbox, and Slack. Use when user says 'daily prep', 'start my day', 'morning briefing', 'what's on my plate today', or asks where to begin. Also fires if user is starting work and asks what needs attention." Edge cases expand the net — "where do I begin?" is a valid trigger that a narrow description misses.
Great description? Now make sure the whole skill actually works.
Step 7 — Test & Debug Your Skill

Testing Methodology

Skills need a test loop like any other code. For each skill, maintain a small set of test prompts in evals/ that covers triggering, execution, and output format.

my-skill/ └── evals/ └── test_cases.md # Trigger Tests - "run the pipeline" → should fire - "analyze this data" → should fire - "what's 2+2" → should NOT fire # Execution Tests - does each step complete? - does it handle missing inputs? # Output Tests - does output match the template? - does it degrade gracefully on failure?

Test sub-skills independently before testing the orchestrator. A broken sub-skill is much harder to diagnose from inside a chain.

Iteration loop: Edit description → test trigger → edit body → test execution → edit output format → test output. Don't skip steps.

Debugging Under-Triggering

If your skill isn't firing when it should, work through this sequence:

1

Check description coverage

Does your description include the actual phrase the user said? Add synonyms and alternate phrasings explicitly.

2

Check for ambiguity with other skills

If two skills could match the same phrase, Claude picks one. Make each description distinct — overlap causes misfires.

3

Add the missed phrase explicitly

If the user's phrasing was reasonable but not covered, add it to the description. Don't rely on Claude to infer it.

4

Check SKILL.md body length

A skill over 500 lines may load slowly or incompletely. Profile which sections are heavy and offload into references.

5

Check sub-skill outputs

If the skill fires but produces wrong output, run each sub-skill in isolation. Identify exactly which step is failing before editing the orchestrator.

Skill works? Now unlock the next level with hooks and memory.
Step 8 — Power Patterns: Hooks & Memory

Hooks: Automated Behaviors

Skills trigger on user intent. Hooks trigger on system events — before/after tool calls, on session start, etc. Together they unlock always-on behaviors that don't require the user to ask.

# settings.json — hook configuration { "hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "echo 'Running bash command'" }] }], "PostToolUse": [{ "matcher": "Write", "hooks": [{ "type": "command", "command": "./scripts/lint.sh" }] }] } }

Skill + Hook combo: Use a skill for the core workflow, and a hook to enforce pre/post conditions automatically. Example: a deploy skill paired with a PostToolUse hook that always runs your test suite after any file write.

When user says "from now on, whenever X" — that's a hook, not a skill. Use the update-config skill to wire it into settings.json so the harness enforces it automatically — not Claude's memory, which resets between sessions.

Memory: Cross-Session Persistence

Skills are stateless — each invocation starts fresh. Memory is the bridge across sessions. Skills can explicitly read and write memory files to maintain state between conversations.

## Step 5: Save State (end of skill) Write a memory entry for anything that should persist beyond this session: - Decisions made + rationale this run - User preferences observed - Next steps to pick up on next session - Pointers to external resources found # Memory file format --- name: pipeline-decisions description: Key decisions from last run type: project --- [content here]

Four memory types your skill can write:

  • user — role, preferences, expertise level
  • project — goals, deadlines, key decisions
  • feedback — corrections and validated approaches
  • reference — pointers to external systems
What NOT to save: code patterns, file paths, git history, or ephemeral task state. Memory is for non-obvious facts that won't be derivable from reading the codebase later.
Last step — make sure you're reaching for the right tool in the first place.
Step 9 — Choose the Right Tool

Skill vs. Hook vs. CLAUDE.md vs. Just a Prompt

Skills are powerful but not always the right choice. Use this table to make the call before you start building.

Situation Right tool Why
Multi-step workflow triggered by user intent
"triage my inbox", "run daily prep", "prep me for my call"		 Skill Skills are built for intent-triggered workflows. The description controls when it fires; the body controls what it does.
Always-on behavior tied to a system event
"after every file write, run lint", "log every bash command"		 Hook Hooks fire on tool events, not user intent. Enforced by the harness — no user prompt needed, and no risk of Claude forgetting between sessions.
Standing instruction that should always apply
"always respond in Spanish", "never commit without asking first"		 CLAUDE.md CLAUDE.md is always loaded into context. Use it for instructions that should shape every response in the project — not just when a skill fires.
One-off task with no reuse value
"rename all these files", "summarize this document"		 Just prompt Don't build a skill for every task. If you'll only do it once and there's no complex workflow to encode, a direct prompt is faster and simpler.
Repeated workflow with variable inputs each time
"analyze this dataset", "prep me for this specific call"		 Skill Skills handle variability well — they read current context rather than relying on hardcoded values. Right fit when the workflow is stable but the inputs change.
Workflow that needs automatic enforcement too
"run tests after every deploy", "validate schema after every write"		 Skill + Hook Compose them. The skill handles the intentional workflow; the hook handles enforcement that should happen regardless of which code path triggered the file write.
Scripts If subagents keep writing the same helper script from scratch, bundle it in scripts/. It costs zero context tokens and every future run benefits.
Large References For reference files over 300 lines, add a table of contents at the top so Claude can navigate efficiently without reading the whole thing.
Error Handling If a sub-skill fails, still run the others. Note the failure in output. A partial result beats no result — design orchestrators to degrade gracefully.
Triggering Under-triggering is the #1 skill problem. Cast a wide net with trigger phrases, then use exclusion notes to narrow if needed.
Memory vs. State Don't save ephemeral task state to memory. Memory is for non-obvious, durable facts — things that won't be derivable from reading the code or git history later.

Skills or Projects?
Here's How to Choose

Both help Claude work better for you. Skills teach Claude how to do something. Projects give Claude something to know. Once you see the difference, it's obvious when to use which.

What's the Difference?

📋 A Skill is a Playbook

A skill is a saved set of steps — a workflow Claude follows every time you trigger it. It answers: "How should Claude do this?"

📋

Like an SOP. You write the steps once. Any time you say "prep me for my call," Claude runs the same checklist — every time.

  • Lives in Claude Code (.claude/skills/)
  • Fires when you say the right phrase
  • Contains step-by-step workflow instructions
  • Best for things you do repeatedly

📚 A Project is a Knowledge Base

A Project (in Claude.ai) is a workspace that gives Claude background knowledge — always active for every conversation you have there. It answers: "What should Claude know?"

📚

Like an onboarding folder. Load in your product docs, tone guidelines, and FAQs once. Claude knows them in every conversation in that project.

  • Lives in Claude.ai workspace
  • Active for every conversation in the project
  • Contains uploaded docs + a system prompt
  • Best for domain knowledge & persistent context
Not sure which to reach for? Here are three quick scenarios.
Quick Decision Guide
When you want Claude to… Use this
Follow the same steps every time ("triage my inbox", "prep me for my call") 📋 Skill
Always know your product, company context, or domain — without you re-explaining it 📚 Project
Have both: structured workflows AND deep background knowledge ✨ Both
They work great together.
Using Them Together

Example: Sales workflow

You create a Project in Claude.ai called "Posit Sales." You upload your ICP definition, product one-pager, competitive battlecards, and tone guidelines. Claude knows all of this in every conversation.

Then you build a Skill called call-prep. When you say "prep me for my 2pm," Claude runs the structured call-prep workflow — pulling in your calendar, researching the prospect, and building a brief.

💡

The Project provides the context (what to know). The Skill provides the workflow (what to do with it). Together, they're more powerful than either alone.

Quick rule Skills = what to DO. Projects = what to KNOW. When in doubt, ask yourself: "Is this a workflow or a knowledge base?"
Projects are Claude.ai Projects live in the Claude.ai app, not Claude Code. If you use the CLI, use CLAUDE.md for persistent context instead.
Start simple You don't need both right away. Write a skill for your most repeated workflow first, then add a Project once you have docs worth loading.

Anti-Patterns to Avoid
❌ Using Skills for Knowledge

Don't pack static reference material (product docs, FAQs, pricing tables) into a SKILL.md body. Skills are for workflows — they're loaded on intent and are not always available.

Fix: Put domain knowledge in a Project (Claude.ai) or in a references/ file that your Skill loads on demand. Keep SKILL.md under 500 lines.
❌ Using Projects for Workflows

Don't define step-by-step procedures in a Project's system prompt. Long procedural prompts in Projects are brittle — they don't modularize, version, or reuse well.

Fix: Move the workflow into a Skill. Reference it from your Project's system prompt with something like: "For call prep, follow the call-prep skill."
